  리뷰 결과 (심각도 순)

  ---
  P0 — 설계 오류 (잘못 이해한 부분)

  1. 피드백 "n개 피드백" 혼선 — 해석을 내려야 함

  README에 "하나의 대화에는 서로 다른 사용자가 생성한 n개 피드백? 대화가 
  공유되진 않을텐데" 라고 물음표로 남겨뒀는데, 이건 요구사항을 잘못 읽은 게
  아니라 요구사항 자체의 모순처럼 보이는 것에 대한 해석이 필요한 상황입니다.

  요구사항의 구조를 보면:
  - "각 사용자는 자신이 생성한 대화에만 피드백 생성 가능"
  - "관리자는 모든 대화에 피드백 생성 가능"
  - "하나의 대화에 서로 다른 사용자들이 생성한 n개의 피드백이 존재할 수 있다"

  → 즉 n개가 가능한 이유는 관리자가 일반 유저의 대화에도 피드백을 달 수 있기 
  때문입니다. 이 해석을 명확히 하지 않으면 (user_id, chat_id) unique 제약을 잘못
  걸거나 피드백 생성 권한 로직이 틀릴 수 있습니다.

  ---
  2. 회원 기능 후순위 + "userId 임시 처리" → JWT 통합 전략 없음

  계획에 userId는 임시로 처리해두기라고 했는데, 요구사항에서는 회원가입·로그인을
  제외한 모든 요청에 JWT 인증이 필수입니다.

  대화 생성 → 피드백 생성 → 분석 전부 userId 기반 권한 분기가 있는데, 이걸 임시
  처리하면 나중에 JWT 붙일 때 API 인터페이스와 서비스 로직을 다 수정해야 합니다.
  처음부터 SecurityContext에서 userId를 꺼내는 구조를 잡아야 나중에 교체가
  쉽습니다.

  ---
  3. RAG 범위가 3시간 내 현실적으로 불가

  README 설계에 chunking → embedding → BM25 → rerank → 답변생성 전 파이프라인을
  구현하겠다고 했는데, 이 중 BM25는 Spring/Kotlin 생태계에 기본 라이브러리가 
  없습니다. Elasticsearch 설정 혼자만 해도 시간을 다 씁니다.

  한편 요구사항의 원문은 "향후 자사의 대외비 문서를 학습시키고 싶어 합니다" —
  향후입니다. 현재 시연의 핵심 요건은 "API로 AI를 활용할 수 있다"이지, RAG가
  아닙니다. pgvector 기반 시맨틱 서치만으로도 충분히 데모가 가능하고,
  BM25/rerank는 MVP 이후 확장 포인트로 두는 게 맞습니다.

  ---
  P1 — 중요한 누락 (구현 시 막히는 부분)

  4. 스레드 30분 로직 설계가 없음

  대화 기능에서 가장 복잡한 로직이 바로 스레드 생성 기준입니다:
  - 첫 질문이거나 마지막 질문 후 30분 초과 → 새 스레드 생성
  - 마지막 질문 후 30분 이내 → 기존 스레드 유지

  이 로직이 어느 레이어에서 처리될지(Service? 별도 ThreadManager?), DB 쿼리를
  어떻게 쓸지(마지막 chat의 createdAt 조회) 설계가 전혀 없습니다. 구현할 때
  막히는 부분 1순위입니다.

  ---
  5. 스트리밍 응답 설계가 없음

  isStreaming: true일 때 SSE(Server-Sent Events)로 응답해야 하는데, 이를 Spring
  MVC에서 할지 WebFlux로 할지가 중요한 선택입니다. Spring MVC +
  SSE(SseEmitter)로 가능하지만 WebFlux가 더 자연스럽습니다. 기술 스택에 명시가
  없어서 구현하다 흔들릴 수 있습니다.

  ---
  6. "스레드 단위 그룹화" 응답 구조 설계 없음

  대화 목록 조회 응답이 { thread: { id, chats: [...] } }[] 형태인지, { threadId,
  question, answer, ... }[] 형태인지 설계가 없습니다. 요구사항에 "스레드 단위로
  그룹화된 대화의 목록"이라고 명시되어 있으므로, 이 중첩 응답 구조를 미리
  잡아야 합니다.

  ---
  7. RBAC(권한 분리) 구현 방식 설계 없음

  member/admin에 따라 동작이 다른 API가 5개 이상입니다(대화 조회, 피드백
  생성/조회/상태변경, 분석). Spring Security의 @PreAuthorize로 갈지 커스텀
  인터셉터로 갈지 설계가 없습니다.

  ---
  P2 — 보통 (세부 설계 누락)

  8. 페이지네이션·정렬 방식 미결

  대화 목록·피드백 목록 모두 오름/내림차순 정렬 + 페이지네이션 필요. offset
  기반으로 갈지 cursor 기반으로 갈지 미결입니다.

  ---
  9. "하루 동안의 기록" 기준 해석 없음

  분석 API에서 "요청 시점으로부터 하루 동안의 기록" — 지난 24시간인지, 오늘
  00:00~23:59인지 해석을 내려야 합니다. (자연스러운 해석은 지난 24시간이지만
  명시가 필요합니다.)

  ---
  P3 — 낮음 (채점 기준 대응)

  10. 채점 기준에서 중요한 항목들이 README에 없음

  요구사항의 채점 기준 4가지 중:
  - "과제를 어떻게 분석했는가" → 현재 README가 이 역할
  - "AI를 어떻게 활용했는가, 어떤 어려움이 있었나" → 없음
  - "가장 어려웠던 기능 설명" → 없음
  - 구현 양은 배점이 낮음

  AI 활용 과정과 난이도 높은 기능에 대한 설명 섹션을 README에 추가해야 합니다.
  구현보다 설명과 분석이 더 중요한 채점 기준입니다.

  ---
  11. 회사 식별 정보 주의 누락
