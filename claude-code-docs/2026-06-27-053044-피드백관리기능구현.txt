     피드백 관리 기능 구현 계획

     Context

     feedbacks 테이블은 이미 V1__init.sql에 존재하지만 Kotlin 코드가 전혀 없음.
     사용자 요청: MEMBER가 채팅에 thumbs up/down 피드백을 남기고, ADMIN이
     PENDING → APPROVED/REJECTED로 검토하는 워크플로우.
     ddl-auto: validate이므로 새 마이그레이션 없이 엔티티만 기존 스키마에 정확히
     매핑하면 됨.

     인증 주의사항

     JwtAuthenticationFilter는 userId: Long을 principal로 저장
     (UsernamePasswordAuthenticationToken(userId, null, authorities)).
     → authentication.principal as Long으로 userId 추출 (기존
     ChatController.resolveUser()의 auth.name + findByEmail 패턴은 버그임 —
     피드백에선 사용하지 않음).
     → 어드민 체크: authentication.authorities.any { it.authority == 
     "ROLE_ADMIN" }.

     생성할 파일 (5개, 기존 파일 수정 없음)

     1. domain/feedback/entity/Feedback.kt

     package com.chatbot.domain.feedback.entity

     import jakarta.persistence.*
     import java.time.Instant

     @Entity
     @Table(name = "feedbacks")
     class Feedback(
         @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
         val id: Long = 0,
         @Column(name = "user_id", nullable = false) val userId: Long,
         @Column(name = "chat_id", nullable = false) val chatId: Long,
         @Column(name = "is_positive", nullable = false) var isPositive:
     Boolean,
         @Enumerated(EnumType.STRING) @Column(nullable = false) var status:
     FeedbackStatus = FeedbackStatus.PENDING,
         @Column(name = "created_at", nullable = false) val createdAt: Instant =
     Instant.now()
     )

     enum class FeedbackStatus { PENDING, APPROVED, REJECTED }
     - isPositive, status만 var (PATCH 대상)
     - @Column(name = "is_positive") 명시 필수 (기존 패턴 + Hibernate 버전별
     차이 방지)
     - enum은 같은 파일 하단에 위치 (User.kt의 Role enum 패턴 동일)

     4. domain/feedback/service/FeedbackService.kt

     주요 메서드:

     createFeedback(userId, req)
     1. chatRepository.existsById(req.chatId) → 없으면 NoSuchElementException
     (404)
     2. feedbackRepository.existsByUserIdAndChatId(userId, req.chatId) → 있으면
     IllegalArgumentException (400)
     3. 저장 후 FeedbackResponse.from() 반환

     listFeedbacks(userId, isAdmin, pageable)
     - isAdmin → feedbackRepository.findAll(pageable)
     - MEMBER → feedbackRepository.findAllByUserId(userId, pageable)

     updateFeedback(userId, isAdmin, feedbackId, req)
     1. findById → 없으면 404
     2. MEMBER이고 feedback.userId != userId → 403
     3. MEMBER이고 req.status != null → 403 (status 변경 불가)
     4. 역할에 따라 필드 업데이트 후 저장

     deleteFeedback(userId, isAdmin, feedbackId)
     1. findById → 없으면 404
     2. MEMBER이고 feedback.userId != userId → 403
     3. feedbackRepository.delete(feedback)

     - @Transactional 쓰기, @Transactional(readOnly = true) 조회
     - 주입: FeedbackRepository, ChatRepository (chat 존재 확인용)

     auth 추출 패턴:
     val userId = authentication.principal as Long
     val isAdmin = authentication.authorities.any { it.authority == "ROLE_ADMIN"
     }

     SecurityConfig 변경 불필요 — anyRequest().authenticated()가 모든
     엔드포인트에 적용됨.
     GlobalExceptionHandler 변경 불필요 — 기존 핸들러가 모든 예외 처리함.

     재사용할 기존 코드

     - ChatRepository.existsById() — JpaRepository 상속 메서드 (chat 존재 확인)
     - GlobalExceptionHandler — 예외 → HTTP 상태 매핑
     - Page<T> 직렬화 패턴 — ChatController.listChats() 동일 방식

     검증 방법

     1. ./gradlew build — ddl-auto: validate으로 엔티티-스키마 매핑 확인
     2. docker compose up --build 후 curl로 각 엔드포인트 테스트:
       - POST /feedbacks (MEMBER 토큰)
       - GET /feedbacks (MEMBER: 본인 것만, ADMIN: 전체)
       - PATCH /feedbacks/{id} — MEMBER는 isPositive 변경, ADMIN은 status 변경
       - MEMBER가 status 변경 시도 → 403 확인
       - 동일 채팅에 중복 피드백 → 400 확인
       - DELETE /feedbacks/{id} — 본인 것 삭제 후 타인 것 삭제 시도 → 403 확인
